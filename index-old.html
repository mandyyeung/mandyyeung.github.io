How the Internet Works

The internet - it’s become so ubiquitous and indispensable that most of us often take it for granted. But, this is important– how does the magic work? What actually happens when we type in a URL on the browser and hit enter? Let’s take a look.

1. Enter an internet address on your browser

For example, let’s say we want to visit Google. We would type in, google.com.

2. Send the request to the internet

Your computer is connected to the internet via your router (and/or modem) and the service is supplied by your ISP (internet service provider). Your ISP provides multiple servers to store and transfer data- NAP (network access protection) server and a DNS (domain name server). The domain google.com is an easy way to remember how to locate Google on the internet, but it’s actually a layer on top of Google’s numerical IP (internet protocol) address. The DNS translates the URL into the IP address to connect you with the web page. For example, you can reach google.com by typing in http://74.125.224.72/.

3. Connect with the web page

With the IP address uncovered, your browser establishes a connection to the world wide web through HTTP (hypertext transfer protocol). HTTP is the standard protocol for communicating with the web, which is just made up of interconnected hypertext documents. Your request for google.com transverses through the cloud until it reaches google.com’s ISP.

4. Obtain and receive data back to your browser

Once google.com’s ISP processes your request to get the website, it sends data back to your browser, loading all the visual goodies such as the day’s Google doodle and the search input box. Ta-da!

Node.js CLI App to Check Stock Prices

This is a simple command line app that uses node.js and the Markit on Demand API to retrieve stock prices. The goal is to be able to obtain results for multiple tickers through a command like so (the following checks for prices of Apple, Facebook, and Yelp stocks) –

node app.js aapl fb yelp

// FB (Facebook, Inc.) is $78.03.
// AAPL (Apple Inc) is $110.43.
// YELP (Yelp Inc) is $54.7.

To start, let’s create a file called stocks.js. We will create our app.js later to refer to the functions in stocks.js.

We want to make a get request to the Markitondemand API, so let’s make use of the HTTP server and client in Node.js.–

var http = require('http');

The way that the API works, we need to pass in a ticker parameter at the end of this call: http://dev.markitondemand.com/Api/v2/Quote/json?symbol=. Let’s create a function called stockLookup that takes one argument, the ticker, and we’ll use Node’s http.get method to access the API–

function stockLookup(ticker){
  var request = http.get('http://dev.markitondemand.com/Api/v2/Quote/json?symbol=' + ticker), // callback function goes here!
}

Now the http.get method takes a callback function with a response argument. The response is an instance of http.IncomingMessage and is the response to the get request. We want this callback function to receive the JSON data from the API and parse it to return the data in the format of “TICKER (Company Name) is $X.XX.” This is what a sample API call will return–

// http://dev.markitondemand.com/Api/v2/Quote/json?symbol=yelp
{
  "Status": "SUCCESS",
  "Name": "Yelp Inc",
  "Symbol": "YELP",
  "LastPrice": 54.7,
  "Change": 0.460000000000001,
  "ChangePercent": 0.848082595870208,
  "Timestamp": "Wed Dec 31 15:59:00 UTC-05:00 2014",
  "MSDate": 42004.6659722222,
  "MarketCap": 3966789300,
  "Volume": 82182,
  "ChangeYTD": 68.95,
  "ChangePercentYTD": -20.6671501087745,
  "High": 55.54,
  "Low": 54.25,
  "Open": 54.68
}

As you can see, the values we want are “Name” and “LastPrice.” The ticker is already passed in to the original API call. With that information in mind, this is our callback function that logs the formatted result into our console–

// function stockLookup(ticker){ ...,
    function(response){
      var body = '';
      response.on('data', function(chunk){
        body += chunk;
      });
      response.on('end', function(){
        var stock = JSON.parse(body);
        var price = stock.LastPrice;
        var name = stock.Name;
        var result = ticker + '(' + name + ') is $' + price + '.';
        console.log(result);
      });
    }
  )};

module.exports.get = stockLookup;

Notice that Node.js streams emit data events each time they receive a chunk of data, and they emit end when they are finished. We continuously save these chunks of data received into an empty string variable named body, and format the data once we have received all of it. After we close out our stockLookup function, we export it so we can call it in our app.js file. Here, we name the exported function get.

Now, let’s create our app.js file–

var stockPrices = require("./stocks");
stockPrices.get("aapl");
stockPrices.get("fb");
stockPrices.get("yelp");

At this point, if you run node app.js in your console, you should get the stock prices of AAPl, FB, and YELP. However, having to provide the tickers within app.js is not dynamic, and calling stocks one at a time is not DRY. One way to improve our code is to pass in an array of tickers and iterate through, calling stockLookup on each ticker. To do this, we can make use of the global object process. Let’s check out process in the console. In your app.js file, add this line–

console.log(process);

Run node app.js aapl fb yelp in your terminal and you will see a long output with information on the line of code you’re running. Particularly, at the key argv, you will see this:

argv:
[ 'node',
'/[path to your file]/app.js',
'aapl',
'fb',
'yelp' ]

Oh look, argv is an array of your command, and we can retrieve the array of tickers we are looking up by calling process.argv.slice(2), because the tickers start at index 2. This is our updated app.js file–

var stockPrices = require("./stocks2");
var stocks = process.argv.slice(2); // slice from index 2 of the array till the end
stocks.forEach(stockPrices.get);

Much better. Now, just tidy up our stocks.js file by refactoring and adding error messages. This is our final result –

var http = require('http');

// Print results

function printResult(tickerUpcase, name, price){
  var result = tickerUpcase + ' (' + name + ') is $' +  price + '.';
  console.log(result);
}

// Print error messages

function printError(error){
  console.error(error.message);
}

function stockLookup(ticker){
  var request = http.get('http://dev.markitondemand.com/Api/v2/Quote/json?symbol=' + ticker, function(response){
    var body = '';
    response.on('data', function(chunk){
      body += chunk;
      });
      response.on('end', function(){
        var stock = JSON.parse(body);
        var tickerUpcase = ticker.toUpperCase();
        if(!Boolean(stock.Message)){
          try {
            var price = stock.LastPrice;
            var name = stock.Name;
            printResult(tickerUpcase, name, price);
          }
          catch(error){
            printError(error);
          }
        }
        else {
          printError({message: 'There was an error getting the price of ' + tickerUpcase + '.'})
        };
      })
    })

  // Connection Error
  request.on('error', printError);
};

module.exports.get = stockLookup;

Sweet!

Concurrency and Parallelism

Distilled into layman’s terms–

Concurrency When multiple tasks can start, run, and complete in overlapping time periods, through shared resources. Tasks may be, but are not necessarily, running at the same instant.

Example of non-concurrency: 1 server, 1 job queue with multiple jobs. Jobs are serviced to completion one by one, sequentially.

Example of concurrency: 1 server, 2 or more job queues. Server shares time with all the 1st jobs in queues. At any instant, there is only one job being serviced.

Parallelism Don’t be confused by their similar dictionary meanings…concurrency and parallelism are NOT the same things! Parallelism is when multiple tasks are being performed at the same time.

Example of parallelism: 2 or more servers, 1 job queue. Two jobs are performed at the same instant. There is no concurrency because the servers are not sharing time and the 3rd job has to wait until one of the servers free up.

Example of parallelism AND concurrency: 2 or more servers, 2 or more job queues.

Deploying to Github via the Gh-pages Branch

Github offers free web hosting for public repositories through Github pages. This comes in handy when you have an open source project to publish, and you don’t want to go through the hassle of setting up a separate website for it. Here’s how to deploy static websites on Github–

  1. You should have already set up a repo for your project. In the root directory, make sure you have a index.html page with the content you want on it.

  2. On Github.com, select the branch dropdown. Fill in–

gh-pages

to create a branch called gh-pages (you know, Github pages!):
img src

  1. Your Github page url comes in this format: http://username.github.io/repository

  2. For future builds, just push to the gh-pages branch:

git push origin gh-pages


Optional: Use a custom domain name with Github pages

To set a custom domain to point to your Github project page, create a new file in your repository named CNAME. You can create a new file on Github.com by clicking on the plus next to your repo name:

The CNAME file just contains the domain name you want to use with this page, without any protocol identifiers. For example, instead of http://www.yourdomain.com, just type in yourdomain.com in your file.

Next, update the DNS (domain name system) record for your domain name with your provider. Login to the control panel at your domain registrar. Navigate to your domain details and look for a link to your DNS settings. There, you’ll have two options.

If you are using a root domain (yourwebsite.com), set up a A record to point to your Github page. As of today, you can point to this IP address: 192.30.252.153. For the latest information, check out the Github help page.

If you are using a subdomain (sub.yourwebsite.com), set up a CNAME record that points to your Github subdomain (ex. username.github.io).

Don’t worry if your custom domain doesn’t work right away. Just give it some time (up to 24 hours) for the DNS changes to take effect.

Intro to Python

I’ve been learning Python with the help of my mentor at StackExchange, and I’m happy to report that it’s been a lot of fun! At the Flatiron School, we focused a lot on Ruby, and Ruby will always be my first true programming language love, but it’s awesome to explore other languages to continue learning programming from different perspectives.

Python is very similar to Ruby. So far, there are just some syntax and terminology differences, such as:

  • Python does not close methods and conditionals with the ‘end’ keyword
  • Python uses colons to declare the start of an indented block
  • Ruby arrays are Python lists
  • Ruby hashes are Python dictionaries
  • Ruby uses booleans true and false; Python uses True and False
  • To read documentation on the command line, use pydoc, (ri in Ruby)
  • Python’s interactive prompt (Ruby’s irb) is python. Exit irb with the keyword ‘exit’. Use ‘exit()’ in python.

Like Ruby, Python comes pre-installed on Mac computers. To run Python scripts, which end with a .py extension (Ruby’s .rb), run python filename.py. For more help with getting Python up and running on your Mac, look through this blog post from HackerCodex.

Let’s check out Ruby vs. Python through code examples:

If/else statements:

# Ruby

x = 2
if x > 2
  puts 'x is greater than 2'
elsif x > 2
  print 'x is greater than 2'
else
  print 'x is equal to 2'
end

# Python
x = 2
if x > 2:
  print 'x is greater than 2'
elif x < 2:
  print 'x is less than 2'
else:
  print 'x is 2!'

For loops, user input, arrays:

# Ruby
colors = []

for i in 1..3
  puts "Enter a color:"
  color = gets
  colors << color
end

# Python
colors = []

for i in range(3):
  color = raw_input("Enter a color:")
  colors.append(color)

Methods/functions:

# Ruby
def square(x)
  squared = x**2
  puts "#{x} squared is #{squared}"
  squared
end

square(5) # => 25

# Python
def square(x):
  squared = x**2
  print "%d squared is %d." % (x, squared)
  return squared

square(5) # => 25

As you can see, going from Ruby to Python is mostly learning new syntax. They’re both object-oriented programming languages, so the jump is not as crazy as say, going from Ruby to Java, which is something I am looking forward to learn about at this week’s Women Who Code Meetup, Intro to Java!

Blogging With Middleman and Github Pages

I’ve been wanting to start a personal finance blog for the longest time, but I never get to keep one going. Now that I’ve got experience maintaining a blog for more than five months (thanks Flatiron School!!), I feel like I can really do this. Plus, I’ve had an eye on a suitable domain name for the blog for the longest time, and I was just recently able to snatch it up for a reasonable price (a couple months ago, it cost around $1,000 to buy it). My boyfriend claims he came up with the name for me, but I’m pretty sure I came up with it myself. Either way, I am now the proud owner of yeungmoney.com, and I can’t wait to get started.

For this blog, I use WordPress, and I’ve tried Jekyll on another blog, but the former nearly gave me a heart attack when my hosting company suspended my account and I thought I’d lost all my posts, and the latter is good, but too bulky for me. Enter Middleman, a light-weight static-site generator that works well with blogs too. A CMS is nice to use for blogging, but it’s not a must-have for me. I prefer to keep my posts as pages and not use a database for Yeung Money.

Here’s a quick run down on how to get a Middleman blog up and up on Github Pages. It’s super simple but quite useful, and leaves you with a lot of room to customize the static portions of your site.

  1. Install the middleman-blog gem on your computer, via the command line:

~~~ gem install middleman-blog ~~~

  1. Cd into the local directory where you are parking your blog and initialize a project:

~~~ middleman init blogname –template=blog ~~~

This will generate the following:

~~~ blogname |- source |- images #subfolder |- javascripts #subfolder |- stylesheets #subfolder |- 2012-01-01-example-article.html.markdown |- calendar.html.erb |- feed.xml.builder |- index.html.erb |- layout.erb |- tag.html.erb |- .gitignore |- config.rb |- Gemfile |- Gemfile.lock ~~~

  1. Note that there’s an example article markdown file already in your project. Remove it and add your own:

~~~ bundle exec middleman article “My first blog post” ~~~

This generates a blog post file under the source directory. Compose away!

  1. When you’re ready to check it out on your local server, run:

~~~ bundle exec middleman ~~~

and go to http://localhost:4567

  1. To publish your blog on to Github pages, which provides free hosting for static web pages, add the gem to your Gemfile:

~~~ gem ‘middleman-gh-pages’ ~~~

  1. Then, generate a Rakefile:

~~~ touch Rakefile ~~~

  1. Require the gem in the Rakefile:

~~~ require ‘middleman-gh-pages’ ~~~

If you check your rake tasks in your console, you should now see build and publish rake tasks.

  1. Make sure you have committed your changes to git. If you have uncommitted changes, you will get an error running the following command, telling you your repo is “dirty.” To deploy with a dirty repo, use the second command (but why would you want to??):

~~~ bundle exec rake publish bundle exec rake publish ALLOW_DIRTY=true ~~~

  1. Check out your live Github page at GITHUB_USERNAME.github.io/REPO_NAME. To set up a custom domain, check out Github Help Pages.

  2. Make your site pretty, enjoy, have fun, and keep blogging!!

Nested Data Structures

Last week was the first full week of Flatiron School (hard to believe, because I feel like I’ve been doing this for much longer…they were not kidding about the program being “fully immersive”!). We did nested data structures on Wednesday, and I am happy to report that I was much more comfortable with these labs than I was with section 3 of the ruby assessment. Pardon the corniness, but this learning process that I’m going through– even at such early stages– is so beautiful and magical. I can’t tell you when exactly I “got” things like nested arrays within nested hashes within more nested hashes, but I sure remember being stuck on the assessment and feeling like this is going to be a loooong semester of pain. It’s not that anything is easy right now, but at least I know where to start and how to navigate a nested data structure problem when I see one. Avi, Blake, and Steven are so awesome at encouraging us not to give up, telling us that they understand being stumped and feeling like the dumbest person in the world when they were beginning too, and they often talk about how things will eventually just “click” if you keep working at it and trying your best. Well, they are totally right (of course)!

So last week I was working on a lab called “collections practice volume 2.” It was made up of 8 nested array questions. The last 3 or so questions were quite difficult for me, but they were really fun because these labs have become like logic puzzles I am addicted to solving. I love see the green texts when running rspec. I do some major internal fist-pumping when they come up on the screen. Anyway, I digress. So I had to spend a lot of time on these last couple of problems, but I finished the lab at around 5:40PM and felt really happy, because I totally needed a break. It was the end of the day, I was tired, and when the pull request didn’t go through like it normally would, I decided I’d do a manual mindless “fix” instead of investigating the issue- I’d just re-clone the repo and move my work over. Of course that idea never works out when you’re not quite paying attention to what you’re doing, and long story short I lost the work I had just spent hours on. Yes, I understand that’s the whole point of Git to prevent something like this from happening…so now I have this incident to remind myself to commit early and commit often! Okay so long story short, it sucked but I didn’t have time to dwell on my mistake. I gave myself approximately 2 seconds to grieve and then moved on to redo my work.

In a way it was a good exercise, right? Theoretically I should’ve been able to fly through the assignment the second time around since I had already solved it once. That was the case for the first couple of questions, but when I reached the problems I had previously struggled with, I found that I couldn’t remember my solutions exactly, though I recalled my logic and approach. For the problem I struggled with the most, I daresay I came up with an even better solution the second time around. Here’s how I did it.

Problem (in the form of the RSpec test):

describe '#merge_data' do

  # Question 6

  it 'combines two nested data structures into one' do
    expect(merge_data(keys, data)).to eq(merged_data)
  end

end

These were the parameters (again, from the RSpec test):

keys = [ { :first_name => "blake" },
         { :first_name => "ashley"} ]
data = [ { "blake" => {
  :awesomeness => 10,
  :height => "74",
  :last_name => "johnson"},
  "ashley" => {
    :awesomeness => 9,
    :height => 60,
    :last_name => "dubs"} } ]

So the task was to take these two nested arrays, match keys to data based on the name (see how the values of the hash in keys match the keys of the hash in data?), and return a hash that looks like this–

merged_data = [ { :first_name => "blake",
                  :awesomeness => 10,
                  :height => "74",
                  :last_name => "johnson" },
                { :first_name => "ashley",
                  :awesomeness => 9,
                  :height => 60,
                  :last_name => "dubs" } ]

The first thing I did for this problem, to help myself understand it better, was to come up with a way to “hard code” it so I can see what I want my program to do.

def merge_data(arr1,arr2)
  merged =[ ]
  merged << arr1[0].merge(arr2[0]["blake"])
  merged << arr1[1].merge(arr2[0]["ashley"])
end

I created an empty array called merged which is the container for my results. “arr1″ is the keys array and “arr2″ is the data array. I can see that, for my hard code answer, I want to merge the first hash of arr1 with the hash of arr2 with key of the name. So on line 3 of my code, I used arr1[0] to access the first element of the keys (for “blake”). Since arr2 is an array with just one element (the hash), I use arr2[0] to access that hash, and we access hashes by keys, so I followed it with the key, “blake”. The merge method for hashes take two hashes and creates a new one with data from first the hash that’s left of the method, then from the hash to the right. I use the “shovel” method, (the «) to append the result of this merge into my merge hash. After doing the same thing for “ashley” (note that arr1 has two elements and the element for “ashley” is at index 1), I have a draft of what I need to code. Since the last line of the method is returned, I do not have to call merged again.

Ok, now for the fun stuff.

I see that I need to iterate over arr1 (I’m tipped off by seeing both arr1[0] and arr1[1] in my hard code). To do this, I use the .each_index method. Within this iteration, I also want to look through arr2, but I am targeting specific matches at this level. I use the .keys method to iterate through the hash, and for each of these two keys, I want to merge arr2 and arr1 if the key of arr2 matches the first name of arr1. After I finish the nested iterations, I need to return the merged array.

def merge_data(arr1,arr2)
  merged =[]
  arr1.each_index do |i|
    arr2[0].keys.each do |name|
      merged << arr1[i].merge(arr2[0][name]) if name == arr1[i][:first_name]
    end
  end
  merged
end

Ta-da!
